# UnityPy 常用操作参考

本文件给 `unity-reverse-workflow` workflow 提供通用的 UnityPy 操作模式。

## 适用场景
- 想先确认某个 Unity 资源文件里有什么对象类型
- 想抽样导出图片、音频、文本对象
- 想避免一上来就全量导出

## 推荐顺序
1. **先枚举对象类型**
2. **再抽样读取对象名和少量内容**
3. **确认样本正确后再批量导出**

不要跳过第 1 步直接全量导出。

## 模式 1：统计对象类型分布
适合：第一次接触某个 `.assets`、`sharedassets*`、`resources.assets`、`bundle`

目标：
- 看这个文件里到底是图片多、文本多、音频多，还是大多是 GameObject / MonoBehaviour
- 决定下一步该走哪条提取路径

输出至少应包含：
- 总对象数
- 按类型分组后的数量
- 每个高价值类型的少量样本名

高价值类型优先关注：
- `TextAsset`
- `Texture2D`
- `Sprite`
- `AudioClip`
- `MonoBehaviour`
- `AssetBundle`

## 模式 2：抽样读取对象名
适合：
- 想知道命名是否有语义
- 想判断 `Level`、`Story`、`BGM`、`icon`、`portrait` 等是否存在

目标：
- 每个重点类型抽 5-20 个对象名
- 判断命名是否值得继续深挖

注意：
- 对象名为空不代表没价值
- 有些对象的关键信息在脚本字段、容器路径或文本正文里，不在 name 字段

## 模式 3：图片抽样导出
适合：
- `Texture2D`
- `Sprite`

建议流程：
1. 先随机/按名称抽几张
2. 检查导出结果：
   - 是否能正常打开
   - 分辨率是否合理
   - 是否有透明通道
   - 命名是否可用
3. 没问题再批量导出

批量导出前，优先保留：
- 原对象名
- 原文件来源
- 类型（Texture2D / Sprite）

## 模式 4：音频抽样导出
适合：
- `AudioClip`

建议流程：
1. 先抽几个样本
2. 判断：
   - 是否可播放
   - 导出后缀是否合理
   - 是否存在批量失败
3. 只有样本通过后，才做全量导出

如果失败：
- 先记录失败对象名、来源文件、对象类型
- 不要直接说“没有音频”
- 很可能是当前库对编码支持不完整

## 模式 5：文本对象读取
适合：
- `TextAsset`
- 可序列化的文本载体

目标：
- 区分哪些是：
  - 配置表
  - 关卡表
  - 文本池
  - 剧情/任务/线索文本
  - 纯辅助资源

建议做法：
- 先读前几个样本
- 看文件名、对象名、正文结构
- 对二进制正文先只做最小解析

## 模式 6：MonoBehaviour 只做轻量探查
适合：
- 想知道是否存在关卡脚本、模板脚本、配置脚本

目标：
- 先拿到脚本名/类名/显眼字段
- 不要在没有结构信息时硬解释全部字段

优先关注字段关键词：
- `level`
- `stage`
- `mission`
- `story`
- `clue`
- `template`
- `config`
- `audio`
- `sprite`
- `icon`

## 结果应该如何表述
推荐用这种格式：
- `sharedassets1.assets` 中共有 842 个对象，其中 `Texture2D=128`、`Sprite=95`、`TextAsset=17`。
- 抽样后确认 `Texture2D` 多为 UI/icon 资源，可继续批量导出。
- `TextAsset` 中有 3 个对象名明显像关卡/剧情数据，其余更像配置或文本池。
- 当前尚未证明这些文本对象与具体关卡存在稳定引用。

避免写成：
- “看起来像关卡文件，应该就是这个。”
- “我导出了一堆图片，所以已经拿到资源层。”

## 何时该停止继续用 UnityPy 深挖
出现以下情况时，考虑切线：
1. 连续两轮只有对象枚举，没有新增语义入口
2. 导出的都是展示资源，但用户真正要的是关卡逻辑
3. 音频/文本对象能看到，但解码/解释明显受限
4. 更关键的信息已经转移到 Managed / metadata / 运行时类层
