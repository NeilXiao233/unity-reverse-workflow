# 音频提取参考

本文件给 `unity-reverse-workflow` workflow 提供通用音频提取思路。

## 目标
帮助 agent 在 Unity 项目中判断：
- 哪些文件可能包含音频
- 当前工具是否能直接导出
- 导出失败时应如何落盘，而不是误判为无资源

## 优先对象
按优先级检查：
1. `AudioClip`
2. 包含音频命名的 `AssetBundle`
3. 可疑 `TextAsset` / 配置表中的音频路径或事件名
4. Managed / metadata 中的音频系统类名

## 标准流程
### 第一步：确认是否存在 `AudioClip`
不要先猜文件名，先确认对象类型。

### 第二步：抽样导出
只抽少量样本，确认：
- 文件能否正常生成
- 是否能播放
- 文件名是否保留了对象语义

### 第三步：决定是否批量导出
满足以下条件再批量：
- 至少 2-5 个样本导出成功
- 命名或目录结构对后续检索有帮助

### 第四步：记录失败样本
若导出失败，记录：
- 来源文件
- 对象名
- 类型
- 失败现象

这一步很重要，因为失败通常意味着编码/容器支持问题，不等于资源不存在。

## 常见语义桶
导出后可按语义分桶：
- BGM
- UI 音效
- 点击反馈
- 成功/失败音效
- 角色语音
- 场景环境音

分桶依据优先来自：
- 对象名
- 包路径
- 相邻配置文本

不是来自主观猜测。

## 和关卡逆向的关系
音频通常是辅助层，不要默认它能直接闭环到关卡。

只有在以下情况才把它升级为较强线索：
- 命名直接含 level/stage/mission/chapter 标识
- 配置表明确引用了同名音频 key
- managed 字段把关卡对象与音频事件绑定在一起

否则默认只当作资源层信息。

## 推荐输出
- 哪些容器里确实存在 `AudioClip`
- 成功导出多少条
- 是否有命名规律
- 哪些失败，以及失败更像工具限制还是资源损坏
- 当前这些音频与关卡关系是否仍停留在辅助层
